# Security Policy

## Supported Versions

This project is under active development. Security fixes are applied to the default branch (`develop`) and will be included in the next tagged release. Older tags may not receive backported fixes unless explicitly noted.

## Reporting a Vulnerability

If you discover a security vulnerability, **please do not open a public issue**. Instead, report it privately so we can investigate and fix the problem before details are disclosed broadly.

- **Preferred method**: Email the maintainers at `security@example.com` with the subject line `SECURITY: <short description>`.
- Include as much detail as possible:
  - A clear description of the issue and its impact.
  - Steps to reproduce (proof-of-concept if available).
  - Any logs, screenshots, or configuration details that help illustrate the problem.
  - Your environment (OS, Go version, deployment details).

If you are unsure whether an issue is security-sensitive, err on the side of caution and report it privately.

## Response Expectations

We aim to:

- **Acknowledge** your report within **3 business days**.
- **Triage** and begin investigating within **7 business days**.
- **Provide a remediation plan or fix** as soon as reasonably possible, depending on the complexity and impact of the issue.

You will be kept informed of progress and notified when a fix is available. When appropriate, we will also discuss coordinated disclosure timing with you.

## Responsible Disclosure Guidelines

We kindly ask that you:

- Give us a reasonable amount of time to investigate and release a fix before publicly disclosing details.
- Avoid accessing, modifying, or destroying data that does not belong to you.
- Avoid actions that could degrade service for other users (e.g., DoS testing on production systems).
- Respect privacy and applicable laws at all times.

Once a fix is released, we may acknowledge your contribution in release notes or a security advisory, unless you prefer to remain anonymous.

## External penetration tests

Third-party penetration tests are planned on an **annual** basis and **before major launches**, with findings tracked as GitHub issues. See [`docs/PENETRATION_TESTING.md`](docs/PENETRATION_TESTING.md).

## Non‑Security Issues

For non-security bugs, feature requests, or general questions, please use the normal GitHub issue tracker instead of the security contact described above.
